## Цель проекта

Построить агентную систему которая автоматизирует весь цикл найма — от поиска вакансий до получения оффера — и служит флагманским кейсом на портфолио.

**Двойная цель:**
- Получить оффер 150-200к на growth/product позицию
- Задокументировать систему как кейс: "я отнёсся к найму себя как к продукту"

## Агенты

### Агент 1 — HH-парсер
**Вход:** ссылка на вакансию с HH  
**Действие:** дёргает HH API, извлекает данные  
**Выход:** JSON → запись в `vacancies` и `companies`

### Агент 2 — Копирайтер
**Вход:** запись из `vacancies` + контекст из `companies`  
**Действие:** генерирует персонализированное письмо  
**Выход:** текст письма → Telegram тебе на подтверждение  
**Промпт:** см. `.claude/agents/Копирайтер.md`

### Агент 3 — Аналитик трафика
**Вход:** данные Posthog за период  
**Действие:** анализирует воронку, находит узкие места  
**Выход:** JSON → `{"работает": "...", "не_работает": "...", "рекомендация": "..."}`

### Агент 4 — Советник по сайту
**Вход:** аналитика от Агента 3  
**Действие:** формулирует конкретные правки для сайта  
**Выход:** текст рекомендаций → Telegram тебе

### Агент 5 — Аутрич-менеджер
**Вход:** подтверждённое письмо + контакт  
**Действие:** фиксирует отправку, трекает статусы  
**Выход:** обновление статуса в `messages` и `vacancies`

## Триггеры n8n

| Событие | Действие |
|---|---|
| Новая запись в `vacancies` | Запустить агента-копирайтера |
| `contact_clicked` в Posthog | Уведомление в Telegram |
| `case_opened` без `contact_clicked` за 24ч | Пометить как "не дошёл" |
| Подтверждение письма в Telegram | Отправить и зафиксировать в `messages` |
| Нет ответа 3 дня | Follow-up напоминание |
| `hr_replied` | Обновить статус в `vacancies` |
| `interview_scheduled` | Напоминание за день |

## Промежуточные метрики успеха

Система работает если за неделю:
- 3+ ответа из 20 отправленных писем
- 2+ запроса на собес
- Данные в Posthog показывают источники трафика

Система работает если за месяц:
- 1+ оффер получен
- Воронка от письма до собеса < 7 дней
- Конверсия contact_clicked > 10% от pageview

## Слабые места и риски

### 1. n8n — единая точка отказа
**Проблема:** Если Amvera упала или n8n завис — вся система стоит. Нет retry логики, нет алертов.  
**Решение (минимум):** Telegram-уведомление при ошибке воркфлоу через Error Trigger. Проверять здоровье n8n раз в час через Webhook Ping.  
**Решение (норм):** Добавить поле `processed_at` в `vacancies`. n8n при старте проверяет вакансии с `status=new AND processed_at IS NULL` — идемпотентный retry.

### 2. Нет версионирования анализа вакансий
**Проблема:** При переанализе вакансии старый результат теряется. Непонятно когда и с каким промптом был получен результат.  
**Решение:** Таблица `vacancy_analysis` хранит несколько строк на одну вакансию. Добавить `version` или `analyzed_at` + `prompt_key`. Текущий анализ = последняя запись по дате.

### 3. Промпты не версионируются
**Проблема:** Изменил промпт → старые результаты несравнимы с новыми. Нет понимания что изменение дало.  
**Решение (сейчас):** `updated_at` в таблице `prompts` — хотя бы видно когда менялось.  
**Решение (норм):** Добавить таблицу `prompt_history (prompt_key, content, created_at)`. Писать туда при каждом изменении `prompts`.

### 4. Нет связи письмо → результат
**Проблема:** Система не знает какое письмо дало ответ. Аналитика по копирайтингу невозможна.  
**Решение:** Поле `message_id` в `vacancies` (или `replied_to_message_id` в `messages`). При смене статуса на `replied` — фиксировать какое конкретно письмо (UTM?) получило ответ.

### 5. Дубли компаний
**Проблема:** `company_id` — n8n не умеет надёжно матчить существующие компании. Continue on fail — костыль.  
**Решение:** Добавить `UNIQUE(name, site)` на таблицу `companies` + использовать `upsert` в n8n вместо `insert`. Или создать отдельный SQL-индекс по нормализованному `lower(trim(name))`.
