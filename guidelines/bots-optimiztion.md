Проведи аудит TG-парсера и дай конкретные правки.

Стек: Node.js, Supabase, Яндекс Cloud Functions
Источник данных: скрапинг t.me/s/{username} — публичный HTML, без API
Режим работы: cron-запуск, обходит список каналов из Supabase (tg_channels), пишет посты в raw_vacancies

Что проверить:

1. УСТОЙЧИВОСТЬ И ОБРАБОТКА ОШИБОК
   — Каждый HTTP-запрос к t.me/s/{username} обёрнут в try/catch
   — При сетевой ошибке или не-200 статусе: пишем в parser_runs.status = 'error' + error_message, не бросаем исключение наружу
   — Если канал недоступен (приватный, удалён, переименован) — логируем и пропускаем, не останавливаем весь прогон
   — Нет мест где ошибка одного канала убивает обработку остальных

2. ДЕДУПЛИКАЦИЯ
   — При записи в raw_vacancies используется INSERT ... ON CONFLICT (tg_message_id) DO NOTHING
   — tg_message_id извлекается из data-post атрибута HTML-элемента поста
   — Проверь: один и тот же пост не может попасть в таблицу дважды даже при параллельных запусках

3. ПРОИЗВОДИТЕЛЬНОСТЬ
   — Задержка между запросами страниц (800мс) вынесена в константу, не захардкожена в коде
   — Нет лишних await внутри циклов которые можно распараллелить
   — При первом запуске канала (last_post_id = null) парсер не тянет больше чем depth_days дней — проверь что условие выхода из пагинации работает корректно
   — Нет утечек памяти при обходе большого количества страниц

4. ЛОГИРОВАНИЕ В parser_runs
   — Перед стартом обхода канала: создаём запись с started_at
   — После завершения: обновляем finished_at, posts_collected, status
   — При ошибке: status = 'error', error_message содержит читаемое сообщение (не stack trace)
   — last_run_at в tg_channels обновляется после каждого запуска канала

5. ТАЙМАУТ CLOUD FUNCTION (5 минут)
   — Есть ли защита от превышения лимита при большом канале?
   — Рекомендация: добавить счётчик времени старта, при приближении к 4:30 — остановить пагинацию, сохранить текущий last_post_id, завершить функцию штатно
   — Проверь что при принудительном завершении по таймауту last_post_id не сбрасывается

6. ФОРМАТ ВЫВОДА В raw_vacancies
   — Все обязательные поля присутствуют: tg_message_id, raw_text, post_url, channel_username, posted_at, source = 'tg'
   — posted_at пишется в UTC, не в локальном времени сервера
   — raw_text не обрезается и не модифицируется — пишем как есть, нормализация на стороне n8n
   — post_url формируется как https://t.me/{username}/{post_id}
